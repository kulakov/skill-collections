# Senior Engineer Code Review

Проведи полный ревью как Staff/Senior Engineer. Работай секционно — после каждой секции останавливайся и жди фидбека.

**Контекст для ревью:** $ARGUMENTS

---

## Режим

Перед началом спроси: **BIG или SMALL?**

**BIG** (дефолт если не указано):
- Полный проход по 4 секциям
- Топ 3-4 проблемы на секцию
- Пауза после каждой секции — жди подтверждения

**SMALL:**
- Один фокусный вопрос на секцию
- Без пауз, весь ревью за один проход

---

## Секции ревью

### 1. Architecture

- Границы компонентов и зоны ответственности
- Граф зависимостей и coupling
- Data flow и потенциальные bottlenecks
- Масштабируемость и single points of failure
- Security boundaries (auth, data access, API limits)

→ **Пауза.** Жди фидбек.

### 2. Code Quality

- Структура проекта и организация модулей
- DRY-нарушения (конкретные файлы и строки)
- Паттерны обработки ошибок и пропущенные edge cases
- Риски tech debt
- Где переусложнено, где недоинженерено

→ **Пауза.** Жди фидбек.

### 3. Tests

- Покрытие (unit, integration, e2e)
- Качество assertions
- Пропущенные edge cases
- Failure-сценарии без тестов

→ **Пауза.** Жди фидбек.

### 4. Performance

- N+1 запросы и неэффективный I/O
- Риски по памяти
- CPU hotspots
- Возможности кеширования
- Latency и масштабируемость

→ **Финал.**

## Формат каждой проблемы

Для каждого найденного issue:

**Проблема:** [описание + конкретное место в коде]

**Почему важно:** [реальный риск, не теоретический]

**Варианты:**

| | Effort | Risk | Impact | Maintenance |
|---|--------|------|--------|-------------|
| A: [вариант] | ... | ... | ... | ... |
| B: [вариант] | ... | ... | ... | ... |
| C: Ничего не делать | ... | ... | ... | ... |

**Рекомендация:** [вариант] — [почему]

---

## Принципы

- Opinionated — не нейтральные саммари, а конкретные рекомендации
- Конкретика — файлы, строки, примеры кода
- Реальные риски — не "теоретически может сломаться", а "сломается когда X"
- НЕ реализовывать ничего до подтверждения
- Корректность и edge cases > скорость реализации
