# Skill Doctor — полный аудит системы скиллов

Проведи аудит как Staff Engineer ревьюит систему. Работай секционно — после каждой секции останавливайся и жди фидбека.

## Действие: $ARGUMENTS

Перед началом определи режим по аргументу:

| Аргумент | Режим |
|----------|-------|
| `<name>` (без ключевого слова) | fix — починить конкретный скилл |
| `check <name\|all>` | check — оценить score |
| `fix <name>` | fix — починить конкретный скилл |
| `batch` / `fix all` | batch — массовая починка |
| `usage` | usage — анализ использования |
| `learn` | learn — извлечение триггеров из логов |
| `health` | health — полный чекап системы (все секции) |

---

## Scan Locations

Искать файлы скиллов в этих папках:

1. `~/.claude/commands/*.md` — глобальные команды (исключая `_archive/`)
2. `~/.claude/skills/*/SKILL.md` — сложные скиллы
3. `/Users/lance/lance-claude/.claude/commands/*.md` — проектные команды
4. `/Users/lance/lance-claude/.claude/skills/*/SKILL.md` — проектные сложные скиллы

**Пропускать:** `start-*` (курсовые модули), файлы в `_archive/`.

## Режим: `check`

### `check <name>`

Найти файл скилла, прочитать frontmatter, оценить quality score:

| Score | Критерий |
|-------|----------|
| 0 | Нет frontmatter, или description пустой, или только English title без описания |
| 1 | Есть description на RU, но нет триггерных фраз ("Используй когда...") |
| 2 | Есть триггерные фразы, но нет антитриггеров ("НЕ используй для...") |
| 3 | Полный pushy: RU описание + триггерные фразы + антитриггеры с указанием альтернативных скиллов |

Вывести:
```
/skill-name — Score: 2/3
Текущий description: "..."
Проблема: нет антитриггеров
Рекомендация: добавить "НЕ используй для: X (skill-a), Y (skill-b)"
```

### `check all`

Просканировать все locations, оценить каждый, вывести сводку:
```
Score 3 (pushy):    145 скиллов
Score 2 (no anti):   23 скилла
Score 1 (no triggers): 15 скиллов
Score 0 (broken):     8 скиллов
---
Кандидаты на fix (score < 3): 46 скиллов
```
Затем список скиллов с score < 3, сгруппированных по score.

## Режим: `fix <name>` (default)

Прочитать скилл и сгенерировать pushy description. Работай по секциям.

### Секция 1: Диагностика

1. Найти файл скилла по имени в scan locations
2. Прочитать содержимое целиком
3. Оценить текущий score (0-3)
4. Вывести диагностику:

```
### /skill-name — Score: 1/3

**Что делает:** [1 предложение из анализа содержимого]
**Текущий description:** "..."
**Проблемы:**
- Нет триггерных фраз
- Нет антитриггеров
- Description на английском
```

> **Пауза.** Жди подтверждение перед генерацией.

### Секция 2: Генерация pushy description

1. Прочитать `~/.claude/SKILLS_CATALOG.md` — найти связанные скиллы в той же категории (для антитриггеров)
2. Если есть данные в `~/.claude/skill-usage.jsonl` — найти реальные фразы пользователя, которые привели к вызову этого скилла (join prompt → skill по session_id + timestamp)
3. Сгенерировать по шаблону:

```
"[RU что делает скилл, 1 предложение]. Используй когда пользователь говорит '[фраза1-RU]', '[фраза2-RU]', '[фраза-EN]', '[casual вариант]'... НЕ используй для: [антитриггер1] ([имя-скилла]), [антитриггер2] ([имя-скилла])."
```

**Правила генерации:**
- Description на русском (даже если скилл на английском)
- Триггерные фразы: 3-6 штук, смесь RU + EN + casual
- Если есть реальные фразы из логов — включить их (они проверены практикой)
- Антитриггеры: 2-4 штуки, всегда с указанием альтернативного скилла в скобках
- Сохранить все остальные поля frontmatter (aliases, name, metadata и т.д.)

4. Показать diff:

```
**Было:**
description: "Short English description"

**Стало:**
description: "Полное русское описание. Используй когда... НЕ используй для..."
```

**Почему важно:** без pushy description скилл не триггерится по фразам пользователя, его можно вызвать только явно через `/name`.

| | Effort | Risk | Impact |
|---|--------|------|--------|
| A: Применить новый description | low | low | high — скилл начнёт триггериться |
| B: Отредактировать вручную | med | low | high |
| C: Оставить как есть | zero | zero | скилл остаётся невидимым |

> **Пауза.** Жди подтверждение перед записью.

### Секция 3: Запись

- Если нет frontmatter — добавить `---\ndescription: "..."\n---`
- Если есть frontmatter без description — добавить поле
- Если есть description — заменить
- Подтвердить запись

## Режим: `batch` / `fix all`

Массовый прогон по всем скиллам с score < 3.

### BIG (дефолт):
1. Просканировать все locations, оценить каждый
2. Вывести сводку (как check all)
3. Спросить: "Починить N скиллов? (по 5 за раз, с паузами)"
4. Для каждой пятёрки: диагностика + генерация + diff
5. После каждой пятёрки: "Продолжить? (осталось M)"
6. Итоговая сводка: починено / пропущено / ошибки

### SMALL:
1. Сканировать, оценить
2. Для каждого скилла с score < 3 — сгенерировать и применить без пауз
3. Итоговая сводка в конце

## Режим: `usage`

Анализ использования скиллов из `~/.claude/skill-usage.jsonl`.

### Секция 1: Статистика

1. Прочитать `~/.claude/skill-usage.jsonl`
2. Отфильтровать записи с `"type": "skill"` (не prompt)
3. Агрегировать: сколько раз каждый скилл вызван, за какой период
4. Вывести:

```
### Топ-10 по использованию (за последние N дней)
| Скилл | Вызовов | Последний |
|-------|---------|-----------|
| nanobanana | 12 | 2026-03-08 |
| ...

### Ни разу не использованы (N скиллов)
skill-a, skill-b, skill-c, ...

### Период данных: YYYY-MM-DD — YYYY-MM-DD (N дней)
```

> **Пауза.** Жди фидбек.

### Секция 2: Рекомендации

Для скиллов с 0 использований за 60+ дней:

**Проблема:** N скиллов ни разу не вызваны за 60+ дней.

**Почему важно:** Мёртвые скиллы засоряют каталог и description list, замедляя триггеринг.

| | Effort | Risk | Impact |
|---|--------|------|--------|
| A: Архивировать все | low | med — может понадобиться | чище каталог |
| B: Проверить descriptions (может не триггерятся из-за плохого description) | med | low | скиллы оживут |
| C: Оставить | zero | zero | status quo |

**Рекомендация:** Сначала B (проверить descriptions через `batch`), потом A для тех что всё равно не используются.

## Режим: `learn`

Извлечение реальных триггерных фраз из логов для обогащения descriptions.

### Алгоритм

1. Прочитать `~/.claude/skill-usage.jsonl`
2. Для каждого skill invocation:
   - Найти ближайший prompt в том же session_id с timestamp ≤ skill timestamp
   - Это и есть фраза, которая привела к вызову скилла
3. Сгруппировать: `skill_name → [список реальных фраз пользователя]`
4. Для каждого скилла — сравнить реальные фразы с текущими триггерами в description
5. Вывести:

```
### Обнаруженные триггерные фразы

/nanobanana — 5 вызовов
  Текущие триггеры: "нарисуй", "сгенерируй картинку", ...
  Реальные фразы:
  - "сделай мне иконку для приложения"  ← НЕТ в description
  - "нарисуй обложку"                  ← ЕСТЬ
  - "generate a mockup"                 ← НЕТ в description

  Рекомендация: добавить "сделай иконку", "generate mockup"
```

> **Пауза.** Для каждого скилла с новыми фразами — предложить обновить description.

---

## Режим: `health`

Полный чекап системы скиллов. Проходит все секции последовательно с паузами.

### Секция 1: Description Quality (из check all)
- Сводка scores по всем скиллам
- Список проблемных (score < 3)

> **Пауза.**

### Секция 2: Usage Analysis (из usage)
- Топ-10 и zero-usage скиллы
- Период данных

> **Пауза.**

### Секция 3: Trigger Learning (из learn)
- Реальные фразы vs текущие descriptions
- Пропущенные триггеры

> **Пауза.**

### Секция 4: Archive Candidates
- Скиллы с 0 usage за 60+ дней И score < 2
- Кандидаты на перемещение в `~/.claude/commands/_archive/`
- Для каждого: причина (не используется + плохой description = точно мёртвый)

> **Пауза.**

### Секция 5: Системные проблемы
- Есть ли hooks в `~/.claude/settings.json` (PostToolUse + UserPromptSubmit)?
- Есть ли `skill-usage.jsonl` и не пустой ли?
- Есть ли `_archive/INDEX.md`?
- Есть ли секция "Skill System" в `~/.claude/CLAUDE.md`?
- Дубликаты алиасов между скиллами?
- Конфликты антитриггеров (скилл A отсылает к B, а B не существует)?

> **Финал.** Итоговый вердикт + приоритизированный план действий.

## Эталон pushy description

Пример идеального score 3 (из /nanobanana):

```
"Генерация изображений через Gemini API. Используй когда пользователь просит нарисовать, сгенерировать картинку, сделать мокап, иконку, обложку, портрет, схему-как-картинку, инфографику, фон/подложку для карточки, иллюстрацию для презентации или лендинга. Также для доработки ранее сгенерированных изображений ('дорисуй', 'измени фон', 'добавь деталь'). Для карточек с большим количеством текста — генерируй только подложку/фон этим скиллом, текст делай отдельно в HTML. НЕ используй для: анализа существующих изображений, промптов Midjourney (/l-g-midjourney), апскейла (/l-d-enhance), генерации музыки, создания презентаций (/l-d-slides)."
```

**Что делает его хорошим:**
- RU описание функции (1 предложение)
- Конкретные триггерные фразы пользователя
- Edge cases (дорисуй, измени фон)
- Границы (карточки с текстом — только подложка)
- Антитриггеры с именами альтернативных скиллов
